# s4extract

Распаковка файлов **The Sims 4 `.package`** (формат DBPF) сразу в:

- **`.fbx`** и **`.obj`** — 3D-меши. Поддерживаются **мебель/объекты (MODL/MLOD)**
  и CAS-меши (GEOM).
- **`.png`** — текстуры (декодируются из DDS: DXT1/DXT3/DXT5 и несжатых)
- **Unity-материалы** — `.mat` под **HDRP / URP / Built-in** + `.meta` с GUID
- **Коллайдеры** — автоматический **compound-коллайдер** из выпуклых частей
  (V-HACD), пригодный для динамической физики
- **`.prefab`** — готовый объект: меш + материал + Rigidbody + коллайдеры

Без Sims 4 Studio и без Blender — всё читается напрямую из `.package`.

## Мебель, физика и коллайдеры (важно для RP-проектов)

Мебель в Sims 4 — **статические объекты без костей** (формат MODL/MLOD), поэтому
скелет/риг не нужен и не экспортируется.

Готовой «collision mesh» в Sims нет. Для динамической мебели (Rigidbody:
уронить шкаф, толкнуть под углом, класть предметы внутрь/сверху) в физике Unity
(PhysX) есть жёсткие правила:

- **Non-convex MeshCollider не работает с движущимся Rigidbody** вообще.
- **Convex MeshCollider** работает, но «запечатывает» полости (внутрь ящика
  ничего не положить).

Каноничное решение — **compound collider**: один Rigidbody на родителе + несколько
**выпуклых** дочерних коллайдеров, вместе аппроксимирующих форму с полостями.
Инструмент делает это автоматически через рекурсивную приблизительную выпуклую
декомпозицию с последующим безопасным объединением соседних hull: каждый объект
разбивается на N выпуклых частей, каждая экспортируется отдельным мешем, а в
`.prefab` на неё вешается дочерний
`MeshCollider (Convex)`. Все они под одним Rigidbody → PhysX объединяет их в
составной коллайдер.

> Декомпозиция приблизительная. Для сложной мебели с глубокими полостями
> увеличь `--max-hulls`, либо доводи коллайдеры вручную в Unity.

Перед рекурсивной нарезкой каждый цельный компонент LOD0 теперь анализируется
непосредственно. Если он уже похож на простой выпуклый объект, экспортёр сохраняет
один collider вместо набора фрагментов:

- ориентированный box;
- выпуклое тело вращения (цилиндр, конус, круглый frustum, сфера, полусфера,
  выпуклое основание лампы).

Пирамида, клин и прямоугольный frustum не требуют аппроксимации: если исходный
компонент уже является такой простой выпуклой формой, обычный hull-путь сохраняет
его одной точной низкополигональной частью.

Тела вращения строятся универсальным `convex lathe` по профилю «высота → радиус».
Для распознанных чашек, труб, мисок и полых абажуров строится `hollow lathe`:
12 выпуклых секторов стенки и, если обнаружено закрытое дно, отдельный нижний
collider. Центральная полость остаётся доступной. Неуверенно распознанная или
иная явно вогнутая геометрия передаётся в остаточную convex-декомпозицию.
Параметрические mesh помечаются как сохранённые и не заменяются затем
округлённой capsule в Unity.

Связные render mesh дополнительно сегментируются по подтверждённым вогнутым
рёбрам: это позволяет отделять сшитые ножки/опоры от корпуса, не разрезая при
этом обычный выпуклый box на шесть плоских граней.

Все material/submesh-группы LOD0 (`g00`, `g01`, ...) предварительно собираются
в один физический объект. Это исключает независимые перекрывающиеся наборы
коллайдеров на границах материалов. При повторном экспорте старые collider OBJ
и `.meta` удаляются, чтобы `Fix All Exports` не подхватил устаревшие части.

### Очень крупные модели: быстрые большие блоки

Для LOD0 с **20 000+ треугольников** или **8 000+ вершинами** по умолчанию
включается защитный быстрый режим коллайдеров. Второй порог важен для объектов,
где каждая UV/normal-сшивка создаёт новую вершину: у большого моста может быть
умеренное число треугольников, но десятки тысяч таких вершин. Вместо рекурсивной
нарезки по мелким вогнутым деталям режим за один проход делит объект вдоль его
большой горизонтальной оси (и при необходимости на верхний/нижний слой) на
максимум **12 крупных box-частей**. Это предназначено
для мостов, зданий, железнодорожных платформ и другого окружения: физика получает
крупные игровые куски, а не тысячи точных фрагментов по несколько сантиметров.

В логе это видно как `coarse_boxes`, а в прогрессе — «быстрые крупные
коллайдеры». Пороги можно изменить флагами `--coarse-collider-faces`,
`--coarse-collider-max-parts`, `--coarse-collider-target-faces`; детальный
старый алгоритм возвращается флагом `--no-coarse-colliders`.

## Зачем это

Раньше пайплайн был ручным:

```
.package  --(Sims 4 Studio)-->  .blend + .png  --(Blender)-->  .fbx + материалы Unity
```

Теперь одной командой:

```
.package  --(s4extract)-->  .fbx + .obj + .png + .mat (URP)
```

> ⚠️ Важно: внутри `.package` **нет `.blend`** — там лежит формат GEOM (меш Sims) и
> текстуры DDS. То, что раньше получалось как `.blend`, — это результат конвертации
> Sims 4 Studio. s4extract строит меш напрямую из GEOM и сразу пишет FBX/OBJ.

## Установка

Нужен **Python 3.9+** (на Windows ставь с python.org и **обязательно** отметь
галочку «Add Python to PATH»).

Самый простой путь на Windows — ничего не ставить вручную: `s4extract.bat` и
`s4extract-gui.bat` при первом запуске **сами доустанавливают** `numpy` и
`Pillow` в тот же Python, которым запускаются. Если автоустановка не прошла
(например, нет прав) — запусти **`install.bat`**.

Вручную (любая ОС):

```bash
pip install -r requirements.txt
```

(`Pillow` и `numpy` нужны только для **текстур**. Ядро — разбор `.package`,
меши `.fbx`/`.obj` и Unity `.mat` — работает на чистой стандартной библиотеке.
`numpy` опционален: без него используется чуть более медленный
чисто-питоновский декодер DDS.)

### Ошибка `ModuleNotFoundError: No module named 'numpy'`

Значит, зависимости стоят не в том Python, который запускает `.bat` (на
Windows часто стоит несколько Python). Решения:

1. Просто запусти **`install.bat`** — он поставит пакеты в правильный Python и
   проверит установку. Затем снова перетащи `.package` на `s4extract.bat`.
2. Либо вручную тем же интерпретатором:
   `py -m pip install numpy Pillow`
3. Даже без этих пакетов меши (`.fbx`/`.obj`) теперь извлекаются нормально —
   только текстуры временно сохранятся как `.dds` вместо `.png`.

## Использование (CLI)

```bash
# один файл
python -m s4extract MyCC.package

# несколько файлов
python -m s4extract a.package b.package -o output

# целая папка (рекурсивно ищет все .package)
python -m s4extract C:\Mods\ -o extracted
```

### Опции

| Опция | Что делает |
|-------|------------|
| `-o, --out DIR` | папка вывода (по умолчанию `./extracted`) |
| `--no-obj` | не писать `.obj` |
| `--no-fbx` | не писать `.fbx` |
| `--no-png` | не извлекать текстуры |
| `--no-unity` | не генерировать Unity `.mat` |
| `--no-colliders` | не генерировать выпуклые коллайдеры |
| `--no-prefab` | не генерировать `.prefab` |
| `--static` | сделать prefab статическим (без Rigidbody) |
| `--max-hulls N` | целевой бюджет выпуклых частей (по умолчанию 128); геометрия не удаляется, превышение помечается в отчёте |
| `--no-convex-merge` | отключить безопасное объединение соседних/перекрывающихся hull |
| `--merge-max-inflation F` | допустимая доля пустого объёма после объединения (по умолчанию 0.03) |
| `--merge-contact-epsilon F` | максимальный зазор между кандидатами в единицах модели (по умолчанию 0.002) |
| `--merge-max-deviation-ratio F` | допустимая толщина convex-мостика относительно диагонали объекта (по умолчанию 0.005) |
| `--max-verts-per-hull N` | максимум вершин в объединённом hull (по умолчанию 64) |
| `--coarse-collider-faces N` | при N+ гранях LOD0 перейти на быстрые крупные box-коллайдеры (по умолчанию `20000`) |
| `--coarse-collider-vertices N` | также включить быстрый режим при N+ вершинах LOD0 (по умолчанию `8000`) |
| `--coarse-collider-max-parts N` | максимум крупных блоков в быстром режиме (по умолчанию `12`) |
| `--coarse-collider-target-faces N` | ориентир граней на один крупный блок (по умолчанию `15000`) |
| `--no-coarse-colliders` | отключить быстрый режим и всегда строить детальные коллайдеры |
| `--all-lods` | экспортировать все LOD (в текущем CLI включено по умолчанию; `--no-lods` оставляет только LOD0) |
| `--inspect` | диагностика: показать типы/сжатие/сигнатуры ресурсов и выйти |
| `--pipeline builtin\|urp\|hdrp` | рендер-пайплайн для `.mat` (по умолчанию **builtin**) |
| `--raw` | дополнительно выгрузить все сырые ресурсы (`raw/`) |
| `--json` | вывести отчёт в JSON |
| `--progress` | показывать объект, этап и процент; подходит для долгих FullBuild-экспортов |
| `--no-resume` | не пропускать завершённые объекты и пересобрать всё |
| `--no-linked-fullbuilds` | отключить автоматическую связь соседних `ClientFullBuildN` |
| `--no-game-resource-search` | не искать недостающие TGI в Build/DeltaBuild установленной игры |
| `-q, --quiet` | меньше обычного вывода (прогресс с `--progress` остаётся) |

> **Built-in vs URP vs HDRP:** выбирай тот пайплайн, который **уже стоит в твоём
> проекте**. Материал чужого пайплайна отображается розовым (несовместимые
> шейдеры). По умолчанию генерируется Built-in/Standard; для URP-проекта запускай
> с `--pipeline urp`, для HDRP — с `--pipeline hdrp`.

### Windows: двойной клик

- `s4extract.bat` — перетащи на него `.package` (или запусти из консоли).
- `s4extract-gui.bat` — откроет графическое окно (выбор файлов, чекбоксы, лог).

### Официальные DLC, разделённые на `ClientFullBuildN`

Для официальных файлов Sims 4 не нужно вручную указывать, в каком паке лежат
текстуры. Если перетащить на `s4extract-hdrp.bat` один из файлов вида
`ClientFullBuild0.package`, `ClientFullBuild1.package` и т. п., экспортёр
автоматически находит пронумерованные соседние части в той же папке. Он строит
общий индекс ресурсов по ключу **TGI** (`type + group + instance`), поэтому
меш из `ClientFullBuild0` получает именно те diffuse/normal/specular-текстуры,
на которые ссылается его MATD, даже если они физически находятся в
`ClientFullBuild2`.

Для отсутствующих TGI экспортёр также пытается найти `*FullBuild*` и
`*DeltaBuild*` в корне установленной игры (он определяется по папке `Data`).
Это нужно для общих текстур базовой игры и обновлений. Поиск читает только
индексы паков; сами данные читаются лениво лишь из файла, содержащего нужную
текстуру. В логе выводятся использованные связанные паки и число текстур,
прочитанных из каждого.

Отключить это поведение из командной строки можно флагами
`--no-linked-fullbuilds` и `--no-game-resource-search`.

### Компактный каталог для нейросети

В дополнение к рабочему `catalog_database.json` экспортёр создаёт
`catalog_ai.json`: одну минифицированную строку со строками вида
`[id,internal_name,[x,y,z]]`. Это отдельный файл, поэтому служебные ключи
экспортёра (`_obj_key`, filename, пути, цвета) не расходуют контекст нейросети.

Эталоны `catCondo_1x1_Med` и `catCondo_2x1_High` подтвердили масштаб:
**1 raw export unit = 1 Sims build tile** по X и Z. Поэтому каталог использует
`u:"t10"`: размеры в десятых долях клетки (`10 = 1 клетка`, `20 = 2 клетки`).
Рядом хранится `catalog_ai_settings.json` с `tile_size: 1.0`; его можно
переопределить только для намеренно масштабированного источника.

Запусти `build-ai-catalog.bat` или команду
`python -m s4extract --build-ai-catalog -o extracted`, чтобы пересобрать JSON
без повторного экспорта. Эта отдельная команда один раз просканирует LOD0 OBJ
старых экспортов без размеров; обычная распаковка не делает такой массовый
скан и не замедляется.

### Автоматические COBJ-теги Build/Buy

Для объектов экспортёр читает список тегов из COBJ (он связан с каждым
CATALOG-свотчем по instance ID), объединяет теги всех свотчей и добавляет в
`catalog_database.json` только полезные категории. Поддерживаются и современный
CommonBlock v10+, и старый 16-битный COBJ-формат ранних Stuff Pack (например,
SP01). Например, у проверенной кровати автоматически получаются:

```json
"tags":["bed","bed_double"]
```

Функциональные теги вроде `Func_Sleep`, `Func_Bed` и `Funk_HideAndSeek`
намеренно не попадают в AI-каталог. Цветовые `Color_*` теги не добавляются к
самому объекту, но считываются **по отдельному COBJ-свотчу**: AI получает
точные строки `[номер_свотча,[цвета]]`, например `blue/gray` или
`dark_brown/yellow`. Ручная запись `v` в `catalog_annotations.json` при
необходимости заменяет автоопределённые цвета конкретного свотча.

Из COBJ ProductStyle TGI автоматически распознаются распространённые стили:
`basics`, `contemporary`, `modern`, `queen_anne`, `french_country`,
`gothic_farmhouse`, `cosmolux`, `suburban_contempo`, `industrial`, `shabby`,
`garden`, `vintage`, `mission`, `tudor`, `shotgun`, `double_gallery`. Они
добавляются к ручным `s`-тегам из `catalog_annotations.json`, а не заменяют их.

Если DLC принесёт неизвестный ProductStyle, он не теряется: его raw TGI будет
записан в `style_refs`, а рядом автоматически появится
`catalog_style_overrides.json` с пустой записью. Впиши туда один раз, например,
`["island","tropical"]`, затем запусти `build-ai-catalog.bat`; модели повторно
экспортировать не нужно.

Для исследования новых/неизвестных тегов в большом паке используй
`diagnose-catalog-tags.bat`: перетащи на него `ClientFullBuild0.package` и
отправь `diagnose_catalog_tags_output.txt`. Отчёт покажет известные категории,
функциональные исключения, неизвестные numeric IDs и ProductStyle TGI.

### Прогресс и безопасное продолжение после остановки

`s4extract-hdrp.bat` показывает в консоли текущий объект, его этап
(модели/текстуры/коллайдеры) и процент среди объектов пакета. Тот же вывод
сохраняется в `s4extract_log.txt`.

После **полностью завершённого** экспорта объекта в его папке создаётся
служебный `.s4extract_complete.json`. При следующем запуске такой объект
пропускается, а работа продолжается со следующего. Идентификация основана не
на видимом имени папки (`[0001] object_name`), а на ключе
`имя package + source instance каталога + TGI модели`; поэтому номер в
скобках и само отображаемое имя не создают ложных совпадений. Если запуск был прерван посередине
объекта, остаётся служебный маркер «in progress», поэтому только этот неполный
объект будет собран заново.

Для совместимости со старыми экспортами без маркера скрипт один раз принимает
уже существующий визуальный `.fbx` (либо не-коллайдерный `.obj`) в папке,
найденной через `catalog_database.json` и source instance. Он создаёт маркер и
переходит к следующему объекту; по текстовому имени папки поиск не выполняется.

Если нужно сознательно пересобрать всё, запусти через консоль с `--no-resume`.
Чтобы пересобрать **только один** объект (например, мост, который был прерван
до быстрой нарезки), создай пустой файл `.s4extract_rebuild` в его экспортной
папке. При успешном окончании он будет удалён автоматически.

## Структура вывода

Для `MyCC.package` создаётся папка `extracted/MyCC/`:

```
MyCC/
├── MyCC_mesh00.fbx          # меш (FBX, ASCII 7.4)
├── MyCC_mesh00.obj          # тот же меш в OBJ
├── MyCC_tex00.png           # текстура(ы)
├── MyCC_tex00.png.meta      # Unity .meta (GUID для привязки)
├── MyCC_swatch00_material.mat      # Unity-материал для первой палитры/свотча
├── MyCC_swatch01_material.mat      # материал для следующей палитры, если есть
└── MyCC_swatch00_material.mat.meta
```

### Импорт в Unity

1. Скопируй содержимое папки в `Assets/` твоего URP-проекта.
2. Unity подхватит `.fbx`, `.png`, `.mat` автоматически — материал уже ссылается
   на текстуры через GUID в `.meta`.
3. Запусти `Tools → s4extract → Fix All Exports`.

`Fix All Exports` рассчитан на пакетную обработку 1000+ экспортов. Для каждого
создаваемого `*_READY.prefab` он автоматически запускает оптимизатор коллайдеров:
box-подобные convex `MeshCollider` заменяются ориентированными `BoxCollider`,
длинные детали с близким к круглому сечением — гладкими `CapsuleCollider`,
соседние сонаправленные box объединяются, а заменённые mesh-компоненты удаляются.
Ручной запуск `S4 Collider Optimizer` для каждого префаба не требуется.

## Как это работает (для модификации)

| Модуль | Ответственность |
|--------|-----------------|
| `dbpf.py` | парсер контейнера DBPF 2.0 (заголовок, индекс, чтение ресурсов) |
| `refpack.py` | декомпрессия RefPack/QFS (сжатые ресурсы) |
| `geom.py` | парсер GEOM → вершины/нормали/UV/грани |
| `mesh_export.py` | запись OBJ и ASCII FBX |
| `textures.py` | RLE/DDS → RGBA → PNG (декодеры BC1/BC2/BC3 на numpy) |
| `unity.py` | генерация `.mat` (URP) и `.meta` |
| `extractor.py` | оркестрация всего пайплайна |
| `cli.py` / `gui.py` | интерфейсы |

## Сжатие ресурсов

Поддерживаются все варианты сжатия DBPF:
- **zlib / deflate** (сигнатура `78 9C` / `78 DA`) — современные пакеты The Sims 4;
- **RefPack / QFS** (`xx FB`) — старые пакеты;
- без сжатия.

Тип определяется автоматически по сигнатуре данных.

## LOD

У объектов мебели обычно несколько уровней детализации (MODL = самый детальный
+ MLOD LOD0..LODn). По умолчанию экспортируется **только самый детальный**
вариант, чтобы не плодить наложенные дубли. Нужны все — флаг `--all-lods`.

## Известные ограничения

- **Текстуры в формате RLES/RLE2** (кастомная упаковка S4): если внутри лежит
  обычный DDS — декодируется; полноценный RLES-кодек пока не реализован, в таком
  случае рядом сохраняется `.dds`/`.bin`, который откроет texconv или GIMP.
- **Скелет/веса костей и морфы** в FBX не переносятся (экспортируется статичный
  меш). Для CAS-предметов с риггингом веса нужно донастраивать вручную (как и
  раньше при ручном пайплайне).
- FBX пишется в **ASCII** — Unity и Blender читают его без проблем.
