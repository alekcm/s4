# s4extract

Распаковка файлов **The Sims 4 `.package`** (формат DBPF) сразу в:

- **`.fbx`** и **`.obj`** — 3D-меши. Поддерживаются **мебель/объекты (MODL/MLOD)**
  и CAS-меши (GEOM).
- **`.png`** — текстуры (декодируются из DDS: DXT1/DXT3/DXT5 и несжатых)
- **Unity-материалы** — `.mat` под **HDRP / URP / Built-in** + `.meta` с GUID
- **Коллайдеры** — автоматический **compound-коллайдер** из выпуклых частей
  (V-HACD), пригодный для динамической физики
- **`.prefab`** — готовый объект: меш + материал + Rigidbody + коллайдеры

Без Sims 4 Studio и без Blender — всё читается напрямую из `.package`.

## Мебель, физика и коллайдеры (важно для RP-проектов)

Мебель в Sims 4 — **статические объекты без костей** (формат MODL/MLOD), поэтому
скелет/риг не нужен и не экспортируется.

Готовой «collision mesh» в Sims нет. Для динамической мебели (Rigidbody:
уронить шкаф, толкнуть под углом, класть предметы внутрь/сверху) в физике Unity
(PhysX) есть жёсткие правила:

- **Non-convex MeshCollider не работает с движущимся Rigidbody** вообще.
- **Convex MeshCollider** работает, но «запечатывает» полости (внутрь ящика
  ничего не положить).

Каноничное решение — **compound collider**: один Rigidbody на родителе + несколько
**выпуклых** дочерних коллайдеров, вместе аппроксимирующих форму с полостями.
Инструмент делает это автоматически через **V-HACD** (приблизительная выпуклая
декомпозиция): каждый объект разбивается на N выпуклых частей, каждая
экспортируется отдельным мешем, а в `.prefab` на неё вешается дочерний
`MeshCollider (Convex)`. Все они под одним Rigidbody → PhysX объединяет их в
составной коллайдер.

> Декомпозиция приблизительная. Для сложной мебели с глубокими полостями
> увеличь `--max-hulls`, либо доводи коллайдеры вручную в Unity.

## Зачем это

Раньше пайплайн был ручным:

```
.package  --(Sims 4 Studio)-->  .blend + .png  --(Blender)-->  .fbx + материалы Unity
```

Теперь одной командой:

```
.package  --(s4extract)-->  .fbx + .obj + .png + .mat (URP)
```

> ⚠️ Важно: внутри `.package` **нет `.blend`** — там лежит формат GEOM (меш Sims) и
> текстуры DDS. То, что раньше получалось как `.blend`, — это результат конвертации
> Sims 4 Studio. s4extract строит меш напрямую из GEOM и сразу пишет FBX/OBJ.

## Установка

Нужен **Python 3.9+** (на Windows ставь с python.org и **обязательно** отметь
галочку «Add Python to PATH»).

Самый простой путь на Windows — ничего не ставить вручную: `s4extract.bat` и
`s4extract-gui.bat` при первом запуске **сами доустанавливают** `numpy` и
`Pillow` в тот же Python, которым запускаются. Если автоустановка не прошла
(например, нет прав) — запусти **`install.bat`**.

Вручную (любая ОС):

```bash
pip install -r requirements.txt
```

(`Pillow` и `numpy` нужны только для **текстур**. Ядро — разбор `.package`,
меши `.fbx`/`.obj` и Unity `.mat` — работает на чистой стандартной библиотеке.
`numpy` опционален: без него используется чуть более медленный
чисто-питоновский декодер DDS.)

### Ошибка `ModuleNotFoundError: No module named 'numpy'`

Значит, зависимости стоят не в том Python, который запускает `.bat` (на
Windows часто стоит несколько Python). Решения:

1. Просто запусти **`install.bat`** — он поставит пакеты в правильный Python и
   проверит установку. Затем снова перетащи `.package` на `s4extract.bat`.
2. Либо вручную тем же интерпретатором:
   `py -m pip install numpy Pillow`
3. Даже без этих пакетов меши (`.fbx`/`.obj`) теперь извлекаются нормально —
   только текстуры временно сохранятся как `.dds` вместо `.png`.

## Использование (CLI)

```bash
# один файл
python -m s4extract MyCC.package

# несколько файлов
python -m s4extract a.package b.package -o output

# целая папка (рекурсивно ищет все .package)
python -m s4extract C:\Mods\ -o extracted
```

### Опции

| Опция | Что делает |
|-------|------------|
| `-o, --out DIR` | папка вывода (по умолчанию `./extracted`) |
| `--no-obj` | не писать `.obj` |
| `--no-fbx` | не писать `.fbx` |
| `--no-png` | не извлекать текстуры |
| `--no-unity` | не генерировать Unity `.mat` |
| `--no-colliders` | не генерировать выпуклые коллайдеры |
| `--no-prefab` | не генерировать `.prefab` |
| `--static` | сделать prefab статическим (без Rigidbody) |
| `--max-hulls N` | максимум выпуклых частей на объект (по умолчанию 16) |
| `--all-lods` | экспортировать все LOD (по умолчанию — только самый детальный) |
| `--inspect` | диагностика: показать типы/сжатие/сигнатуры ресурсов и выйти |
| `--pipeline hdrp\|urp\|builtin` | рендер-пайплайн для `.mat` (по умолчанию **hdrp**) |
| `--raw` | дополнительно выгрузить все сырые ресурсы (`raw/`) |
| `--json` | вывести отчёт в JSON |
| `-q, --quiet` | меньше вывода |

> **HDRP vs URP:** выбирай тот, что **уже стоит в твоём проекте**. Материал
> чужого пайплайна отображается розовым (несовместимые шейдеры). По умолчанию
> генерируется HDRP.

### Windows: двойной клик

- `s4extract.bat` — перетащи на него `.package` (или запусти из консоли).
- `s4extract-gui.bat` — откроет графическое окно (выбор файлов, чекбоксы, лог).

## Структура вывода

Для `MyCC.package` создаётся папка `extracted/MyCC/`:

```
MyCC/
├── MyCC_mesh00.fbx          # меш (FBX, ASCII 7.4)
├── MyCC_mesh00.obj          # тот же меш в OBJ
├── MyCC_tex00.png           # текстура(ы)
├── MyCC_tex00.png.meta      # Unity .meta (GUID для привязки)
├── MyCC_material.mat        # Unity URP-материал
└── MyCC_material.mat.meta
```

### Импорт в Unity

1. Скопируй содержимое папки в `Assets/` твоего URP-проекта.
2. Unity подхватит `.fbx`, `.png`, `.mat` автоматически — материал уже ссылается
   на текстуры через GUID в `.meta`.
3. Назначь материал на меш (или используй встроенный в FBX слот).

## Как это работает (для модификации)

| Модуль | Ответственность |
|--------|-----------------|
| `dbpf.py` | парсер контейнера DBPF 2.0 (заголовок, индекс, чтение ресурсов) |
| `refpack.py` | декомпрессия RefPack/QFS (сжатые ресурсы) |
| `geom.py` | парсер GEOM → вершины/нормали/UV/грани |
| `mesh_export.py` | запись OBJ и ASCII FBX |
| `textures.py` | RLE/DDS → RGBA → PNG (декодеры BC1/BC2/BC3 на numpy) |
| `unity.py` | генерация `.mat` (URP) и `.meta` |
| `extractor.py` | оркестрация всего пайплайна |
| `cli.py` / `gui.py` | интерфейсы |

## Сжатие ресурсов

Поддерживаются все варианты сжатия DBPF:
- **zlib / deflate** (сигнатура `78 9C` / `78 DA`) — современные пакеты The Sims 4;
- **RefPack / QFS** (`xx FB`) — старые пакеты;
- без сжатия.

Тип определяется автоматически по сигнатуре данных.

## LOD

У объектов мебели обычно несколько уровней детализации (MODL = самый детальный
+ MLOD LOD0..LODn). По умолчанию экспортируется **только самый детальный**
вариант, чтобы не плодить наложенные дубли. Нужны все — флаг `--all-lods`.

## Известные ограничения

- **Текстуры в формате RLES/RLE2** (кастомная упаковка S4): если внутри лежит
  обычный DDS — декодируется; полноценный RLES-кодек пока не реализован, в таком
  случае рядом сохраняется `.dds`/`.bin`, который откроет texconv или GIMP.
- **Скелет/веса костей и морфы** в FBX не переносятся (экспортируется статичный
  меш). Для CAS-предметов с риггингом веса нужно донастраивать вручную (как и
  раньше при ручном пайплайне).
- FBX пишется в **ASCII** — Unity и Blender читают его без проблем.
